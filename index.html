1. Guildlines

Below, we share the Guildline-enhancement prompt that we used in article

                                                    
- Syntax:
    - Follow YAML syntax, always include required symbols, e.g., colons, hyphens, or quotation marks.
    - Ensure correct indentation and structure for GitHub Actions YAML files.
    - Ensure all used blocks are correctly defined and properly placed.
    - Ensure types are correctly matched, e.g., ''Django>=2.2,<2.3'' must remain a single unit.

- Logic:
    - Ensure all trigger events are correctly translated, e.g., avoid using tags as the sole trigger event.
    - Ensure all execution orders are correctly translated, e.g., deploy blocks should be configured with the `needs` parameter to execute after test blocks.
    - Ensure conditional statements are correctly translated.
    - Ensure all tasks are translated without introducing additional or unrelated tasks.
    - Keep the job structure unchanged and ensure job relationships remain consistent after translation, e.g., between python-version and toxenv.

- Platform Differences:
    - Ensure all Travis CI fields are translated to GitHub Actions fields, e.g., "tox-travis" to "tox-gh-actions". Fields that cannot be translated, e.g., notification and allow_failure, may be discarded.
    - Retain only x64 and arm64; remove all other architectures.
    - Ensure all version numbers are in string format, e.g., Python '3.10'.
    - Explicitly configure fields that are implicitly defaulted in Travis CI, e.g., actions/checkout@v2, actions/setup-python@v2, and test environment setup.
    - Verify that all expressions are properly converted to GitHub Actions syntax, e.g., 'env: TOX_ENV=flake8' to 'TOX_ENV: flake8'.
    - Ensure all referenced dependencies are present.

- Environment:
    - Ensure all actions are currently supported, e.g., actions/cache v3 or higher.
    - Ensure all required tokens are properly configured, e.g., set GITHUB_TOKEN in coveralls blocks.
